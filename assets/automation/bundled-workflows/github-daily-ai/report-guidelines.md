# GitHub AI 日报编辑规范

## 整体结构

1. 标题：`GitHub 每日 AI 新星榜（YYYY-MM-DD）`。
2. 今日速览：用 2 至 4 句话概括榜单里真正出现的技术方向，不写空泛行业判断。
3. Top 10 简榜：只保留排名、项目、当前累计 Star、主语言和创建时间。不要把项目介绍压进表格。
4. 项目详情：按排名逐项展开。
5. 统计口径与限制：放在文末，简明说明时间窗口、查询方式、排序口径、数据缺失和 GitHub 索引延迟。

## 单个项目写法

每个项目使用下面的结构：

```markdown
### 1. owner/repository · ⭐ 123

**一句话定位，直接说明它是什么。**

用 2 至 3 句话说明它解决的问题、主要实现方式和核心能力。先讲用途，再讲技术；遇到 RAG、MCP、推理框架等术语时，要结合项目功能解释，不能只罗列名词。内容必须来自仓库描述、README 或可核验的项目资料，不得把英文 description 生硬直译成中文。

**今日看点：** 用一句话说明它相较其他候选的独特之处，或今天值得关注的具体原因。不要把“Star 高”作为唯一理由。

[查看 GitHub](https://github.com/owner/repository)
```

## 文风约束

- 使用简洁、自然的中文，让不了解项目背景的开发者也能看懂。
- 每个项目的正文建议为 120 至 220 个汉字，信息不足时宁可更短，不得补写不存在的功能。
- 一句话定位、正文和今日看点各自承担不同信息，避免互相重复。
- 不得出现“适合谁使用”字段，也不要讨论目标用户画像。
- 不写“革命性”“颠覆性”“最佳”等无法证实的宣传语。
- 不把 README 的功能列表整段搬进报告，应提炼最关键的 2 至 4 项能力。
- 无法核验的语言、创建时间或 Star 数必须标注“未获取”，不得猜测。

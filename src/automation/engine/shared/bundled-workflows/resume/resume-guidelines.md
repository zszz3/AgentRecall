# 简历写法要点（写每一条项目 bullet 时遵守）

一条好的 bullet = 问题 → 方案 → 量化结果，三段都要“硬”：

1. 先讲问题 / 动机，并用数字体现严重性
   （例：“Redis 幂等结果内存占用高、集群超 90%”）。没有问题的条目只是罗列职责。
2. 方案要有技术深度和取舍，写清你怎么做
   （例：“唯一键 INSERT 抢占决策权 + 状态机 + 短轮询超时降级”），而不是“用了缓存”。
3. 结果必须量化：延迟 / QPS / 百分比 / 金额 / 单量 / 提效
   （例：“4s→100ms 约 30 倍”“退差 20 万+/4056 单”“提效 99%”）。

## 硬性约束

- 动词用“设计 / 落地 / 重构 / 治理”，不用“参与 / 负责”。主语是“我做了什么”。
- 技术名词要具体：写 Redis / TCC / Kafka / Binlog，不写“中间件 / 缓存”。
- 一条只讲一件事；优先突出难点（跨系统一致性、幂等、并发、灰度、性能）。
- 缺指标时用【待补充：指标】占位，不要编造具体数字冒充真实。
- 宁可少写几条精的，也不要一堆空话。

## 对照

- 反例：参与了系统的缓存优化，提升了性能。
- 正例：针对 GetQcLabel 13.5 万 QPS 下 TCC 全量扫描导致 p99 抖动，改为 Redis Hash 按 shopId 分桶 + O(1) 查询，p99 降至 1.6ms。

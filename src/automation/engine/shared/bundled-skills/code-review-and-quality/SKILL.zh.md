# 代码评审与质量

## 概览

这是一个多维度代码评审流程。每个变更在合并前都要经过质量门禁；评审覆盖正确性、可读性、架构、安全和性能五个维度。

批准标准：如果变更明确改善了代码库整体健康度，并且符合项目惯例，就可以批准。不要因为“不是我会写的方式”而阻塞它；完美代码不存在，目标是持续改进。

## 何时使用

- 合并任何 PR 或代码变更前。
- 完成功能实现后。
- 需要评审另一个 Agent、模型或人的代码时。
- 重构现有代码时。
- 修复 bug 后，同时评审修复和回归测试。

## 五轴评审

### 1. 正确性

代码是否做到了它声称要做的事情：

- 是否匹配规格、任务要求和预期行为。
- 是否处理了空值、空集合、边界值等边界条件。
- 是否处理了错误路径，而不只是 happy path。
- 测试是否通过，测试是否真的覆盖目标行为。
- 是否存在 off-by-one、竞态、状态不一致等问题。

### 2. 可读性与简单性

另一个工程师或 Agent 能否不依赖作者解释就理解代码：

- 命名是否清晰，并符合项目惯例。
- 控制流是否直接，是否避免过深嵌套和过度聪明的技巧。
- 相关代码是否组织在一起，模块边界是否清楚。
- 是否能用更少的代码表达同样的行为。
- 抽象是否值得它带来的复杂度。
- 注释是否解释非显然意图，而不是复述代码。
- 是否残留无用变量、兼容占位、已删除注释或死代码。

### 3. 架构

变更是否适合当前系统设计：

- 是否沿用既有模式；如果引入新模式，理由是否充分。
- 模块边界是否干净。
- 是否存在应该提取共享的重复代码。
- 依赖方向是否合理，没有循环依赖。
- 抽象层级是否合适，没有过度设计或过度耦合。

### 4. 安全

检查变更是否引入安全风险：

- 用户输入是否校验和清洗。
- secret 是否没有进入代码、日志和版本控制。
- 需要鉴权/授权的位置是否做了检查。
- SQL 是否参数化，避免字符串拼接。
- 输出是否正确编码，避免 XSS。
- 依赖来源是否可信，是否存在已知漏洞。
- 外部 API、日志、用户内容和配置文件是否被当作不可信输入处理。

### 5. 性能

检查是否引入性能问题：

- 是否出现 N+1 查询。
- 是否存在无界循环或不受限制的数据获取。
- 是否有本应异步却同步执行的操作。
- UI 是否出现不必要重渲染。
- 列表接口是否缺少分页。
- 热路径中是否创建大对象或做重活。

## 变更规模

小而聚焦的变更更容易评审、合并和回滚：

```text
约 100 行变更：好，适合一次评审。
约 300 行变更：如果是单一逻辑变更，可以接受。
约 1000 行变更：太大，应该拆分。
```

一个变更应该是自包含、只解决一件事、带相关测试，并且提交后系统仍能工作。重构和新增行为通常应该分开；少量变量命名清理可以按评审者判断合并。

## 评审流程

1. 先理解上下文：这个变更要解决什么问题，实现什么规格，预期行为是什么。
2. 先看测试：是否存在测试，是否测行为而不是实现细节，边界和回归是否覆盖。
3. 再看实现：逐文件用五轴检查正确性、可读性、架构、安全和性能。
4. 给发现分级：未加前缀通常表示必须改；`Critical` 阻塞合并；`Nit` 是可忽略的小建议；`Optional`/`Consider` 是建议；`FYI` 只是信息。
5. 验证验证过程：确认跑了哪些测试、构建是否通过、UI 是否有截图或前后对比、手动验证是否清楚。

## 多模型评审模式

可以让不同模型承担不同视角：模型 A 写代码，模型 B 评审正确性和架构，模型 A 处理反馈，最后由人做最终判断。不同模型的盲区不同，这样更容易捕捉单模型遗漏的问题。

## 死代码卫生

任何重构或实现后都要检查孤立代码：

1. 找到现在不可达或未使用的代码。
2. 明确列出来。
3. 删除前先确认，尤其是不确定是否仍被外部使用时。

不要留下死代码；它会误导之后的人和 Agent。

## 评审速度

慢评审会阻塞整个团队。一般要求一个工作日内响应，理想情况是请求到达后尽快给出第一轮反馈。大型变更应该要求拆分，而不是硬着头皮评一个巨大变更集。

## 分歧处理

处理争议时按优先级判断：

1. 技术事实和数据高于意见。
2. 样式指南是样式问题的权威。
3. 软件设计按工程原则评估，而不是个人偏好。
4. 代码库一致性可以接受，但前提是不降低整体健康度。

不要接受“以后再清理”。如果不是明确紧急情况，清理应该在合并前完成。

## 诚实评审

- 不要橡皮图章式 “LGTM”。
- 真实问题不要弱化。
- 能量化的问题尽量量化。
- 对明显有问题的方案直接指出并给替代方向。
- 如果作者有更完整上下文并决定覆盖你的建议，要接受这个决定；评论代码，不评论人。

## 依赖纪律

新增依赖前要问：

- 现有技术栈是否已经能解决。
- 依赖体积和 bundle 影响如何。
- 是否活跃维护。
- 是否有已知漏洞。
- license 是否兼容。

优先使用标准库和现有工具。每个依赖都是长期负担。

## 常见误区

| 误区 | 现实 |
| --- | --- |
| 能跑就够了 | 不可读、不安全或架构错误的代码会持续制造债务。 |
| 我写的，所以我知道没问题 | 作者容易忽略自己的假设。 |
| 以后再清理 | 以后通常不会来。评审就是质量门禁。 |
| AI 生成代码应该没问题 | AI 代码更需要仔细审查，因为它经常自信但错误。 |
| 测试通过就好了 | 测试必要但不充分，不能替代架构、安全和可读性评审。 |

## 验证

评审结束前确认：

- Critical 问题已解决。
- Important 问题已解决，或明确记录了延期理由。
- 测试通过。
- 构建通过。
- 验证说明清楚记录了改了什么、怎么验证。

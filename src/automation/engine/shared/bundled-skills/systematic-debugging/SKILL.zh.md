# 系统化调试

## 概览

随机尝试修复会浪费时间，也容易引入新问题。快速补丁经常只是掩盖症状。

核心原则：**先找到根因，再修复。** 只处理表象就是失败。

## 铁律

```text
没有根因调查，就不能开始修复。
```

如果还没有完成第一阶段，就不要提出修复方案。

## 何时使用

任何技术问题都应该使用：

- 测试失败。
- 生产 bug。
- 行为不符合预期。
- 性能问题。
- 构建失败。
- 集成问题。

尤其是在时间压力下、看起来“改一下就好”、已经尝试过多次修复、上一次修复无效，或者你还没有完全理解问题时，更应该坚持这个流程。

## 四个阶段

### 阶段 1：根因调查

在做任何修复之前，先完成这些动作：

1. 仔细阅读错误信息和 warning，包括堆栈、行号、文件路径和错误码。
2. 稳定复现问题，记录准确步骤；不能复现时继续收集证据，不要猜。
3. 检查最近变更，包括 git diff、提交、依赖、配置和环境差异。
4. 对多组件系统添加诊断日志，确认数据在每个边界的输入、输出、配置和状态。
5. 沿调用链回溯错误值的来源，找到最初触发点，而不是在报错位置补丁。

配套的 `root-cause-tracing.md` 详细说明了如何从深层堆栈一路向上追踪根因。

### 阶段 2：模式分析

先找出正常工作的模式，再比较差异：

- 在同一代码库里找到类似的正常实现。
- 如果参考实现存在，完整阅读，不要只扫几行。
- 列出正常实现和异常实现之间的所有差异。
- 理解依赖项、配置、环境和隐含假设。

### 阶段 3：假设与验证

使用科学方法：

1. 写下一个明确假设：我认为根因是 X，因为 Y。
2. 用最小变更验证这个假设，一次只改变一个变量。
3. 验证结果：成功才进入实现；失败就形成新的假设。
4. 不理解时直接承认“不理解”，继续研究或求助，不要假装知道。

### 阶段 4：实现

修复根因，而不是修补症状：

1. 先创建最小失败用例，最好是自动化测试。
2. 只做一个针对根因的修复，不夹带重构和顺手改动。
3. 验证测试通过、其他测试没坏、问题确实解决。
4. 如果修复无效，停止叠加补丁，回到调查阶段。
5. 如果已经连续 3 次修复失败，要开始质疑架构或模式本身，而不是继续硬改。

## 红旗

出现这些想法时，马上停下来回到根因调查：

- “先快速修一下，之后再查。”
- “先改 X 试试看。”
- “一次改几个地方一起跑。”
- “不用测试，我手动验证。”
- “大概是 X，直接修。”
- “我不完全理解，但这个可能行。”
- “再试一次修复。”，尤其是已经失败两次以后。

## 常见借口

| 借口 | 现实 |
| --- | --- |
| 问题很简单，不需要流程 | 简单问题也有根因，流程在小问题上反而很快。 |
| 很急，没时间调查 | 系统化调试通常比猜测式修改更快。 |
| 先试一下再说 | 第一次修复会决定节奏，开始就要做对。 |
| 之后再补测试 | 没有测试的修复很难留下来。 |
| 一次修多个地方更省时间 | 你无法知道哪一个变化真正有效，还会制造新问题。 |
